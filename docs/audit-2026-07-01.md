**Audit date**: 2026-07-01
**Scope**: full repo (backend + frontend + docs + email templates + one-shot scripts)
**Method**: three parallel read-only surveys (repo hygiene / backend correctness+perf / frontend+emails), followed by targeted verification

## How to read this document

Every finding is ranked **HIGH / MEDIUM / LOW** with a specific `file:line` reference so items are directly actionable. A finding is HIGH only when it (a) impacts money, subscriber trust, or user data; (b) already caused a live incident or has a plausible near-term path to one; or (c) creates recurring egress/cost pressure.

Nothing in this report is fixed. The final section groups everything into a suggested sequence — quick wins, sprint-worthy, and deferrable.

**Why the Maps key matters even though `NEXT_PUBLIC_*` is client-visible by design**: the key ships in every user's browser regardless; the risk is that the *committed* value in a public repo can't be rotated without a new build, and it bypasses the Cloud Run env-var flow that the API key uses. A referrer-restricted key on the correct domain is the mitigation today; the moment restrictions drift, the key is exposed until someone remembers to rebuild.

**Why the webhook secret is HIGH**: with the secret unset in prod, anyone POSTing to `/api/v1/webhooks/brevo/` can mutate `BroadcastRecipient` engagement fields and trigger the auto-deactivate-after-3-hard-bounces logic (Sprint 8.5/8.6) against arbitrary subscribers — a silent list-poisoning vector.

**Why donation enumeration is HIGH**: `order_id` is `SJKT-DON-YYYYMMDD-XXXXXX` (6 hex = ~16.7 M/day). Trivially enumerable, and the endpoint returns donor name + amount + status. The URL also lives in the Toyyib return-URL so it leaks via `Referer` to any third-party asset on the thank-you page.

**On the broadcast race**: the incident that triggered Sprint 23's "Recovery Cut" (CLAUDE.md line 392, 4 duplicate broadcast rows) is guarded by a `filter().exists()` check-then-write. If two Cloud Scheduler triggers land within the sub-second check window (network jitter, retry policy, human `gcloud run jobs execute`), the duplicate re-appears. A `UniqueConstraint(fields=["kind", "coverage_start_date", "coverage_end_date"], condition=Q(status__in=[SENT, SENDING]))` closes it at the storage layer.

**On the orphan janitor**: the comment "orphans get janitored later" in `services.py:44-48` is a lie by omission — no janitor command exists in the repo. Every rejected suggestion that had a staged image leaves a file in Supabase Storage. Growth is slow (2-5 per week today) but monotonic and never GC'd.

## Recommended sequencing

### Quick wins — under 1 hour each, low blast radius

1. **Rotate the Google Maps API key and move it to a build-arg**. `frontend/Dockerfile`. Add `ARG NEXT_PUBLIC_GOOGLE_MAPS_API_KEY` and pass via `gcloud run deploy --build-env-vars`. Referrer-restrict to `*.tamilschool.org` at Google Cloud Console.
2. **Require `BREVO_WEBHOOK_SECRET` at prod boot**. `backend/sjktconnect/settings/production.py` — raise on missing, like `SECRET_KEY`.
3. **Add `db_index=True` on the 5 hot-path fields**. One migration per app; each fits under 20 lines.
4. **Delete `StateFilter.tsx` + `MeetingReportsList.tsx`**. Verified dead. ~350 LOC gone.
5. **Add `app/[locale]/error.tsx` + `app/[locale]/not-found.tsx` root boundaries**. Two files, ~50 lines each. Instant DX + user-facing improvement.
6. **Add ▲/▼ glyph to `EnrolmentTrend` trend indicator**. A11y compliance in one line.
7. **Retire `URGENT_ALERT_REQUIRE_REVIEW` branch OR update CLAUDE.md**. Whichever is intended.

### Sprint-worthy — owner picks priority

1. **Fix the image N+1 in `SchoolListSerializer` + nested `Constituency/DUN`** — biggest egress waster on public routes. Options: (a) `Prefetch` with a filtered queryset caching only primary images, (b) denormalise a `School.hero_image_url` field and skip the join.
2. **Migrate `donation_status` to UUID lookup + add rate limit + drop from URL to POST body**. Removes enumeration.
3. **Write an image-orphan janitor command** — scan `Suggestion` rows deleted or rejected > 7 days ago where the file still exists in Supabase Storage. Wire to a weekly Cloud Run job. Also close the "orphans get janitored later" fictional promise.
4. **Add DB `UniqueConstraint` on `Broadcast (kind, coverage_start_date, coverage_end_date)` where `status IN (SENT, SENDING)`**. Closes the 2026-05-02 duplicate race at the storage layer.
5. **Plain-text alternatives for the 5 bulk email templates + inline the urgent template's CSS**. Improves deliverability across Gmail mobile, Yahoo, Outlook.
6. **Sprint 32 test-debt catch-up** — already flagged in CLAUDE.md. Covers `admin_image_upload_view`, `import_enrolment_snapshots`, `SchoolEnrolmentSnapshot`, `EnrolmentTrend`, pending-moderation-count serializer.
7. **Mobile-viewport pass at 375 px** — fix `SchoolImage` thumbnail overflow, `SchoolProfile` StatCard grid, `SupportSchoolCard` name clip, `EnrolmentTrend` label collision. Bundle as one small "responsive polish" pass.

## Assumptions and caveats

- No dynamic profiling was run — perf findings are static-analysis inference. The image N+1s specifically would benefit from a 60-second `django-silk` capture on a real Constituency detail load to size the impact precisely.
- Email rendering was audited by reading source, not by sending to real Gmail / Outlook / Yahoo Mail. A one-time `Litmus` or `Email on Acid` pass would validate the `<style>`-block / `<div>`-root findings before committing to a template rewrite.
- The `broadcast.text_content` gap (do any composers write it?) needs a grep-verify before it becomes an action item.
- Repo hygiene was assessed against `git ls-files`. Untracked local scratch under `/scratchpad/` or shell history is out of scope.
