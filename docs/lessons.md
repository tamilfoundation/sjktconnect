# Lessons Learned

Cross-cutting lessons from SJK(T) Connect development. Project-specific decisions live in CLAUDE.md.

- **Measure layout dimensions before adjusting them**: when an owner asks "make X the same height as Y", use Playwright's `getBoundingClientRect()` to read both heights *first*. Sprint 32 burned 5 deploys + 6 iterations on a chart-height question because I was guessing pixel deltas from screenshots. The actual difference was 10px — would have been ONE deploy if I'd measured first. (Sprint 32)
- **Background-then-edit creates stale builds**: `gcloud run deploy --source .` packages the local directory at command-start time, not at any specific git revision. If you start a background deploy then edit the source for the *next* change, the running deploy captures the OLD source — only the next deploy picks up your edits. Sprint 32 chained 3 of these on the support card and confused the live page state for 20 min. Pattern: queue all small edits into ONE commit + ONE deploy, OR wait for each deploy to land before the next edit. (Sprint 32)
- **ISR-bust via env-var doesn't rebuild the image**: `gcloud run services update --update-env-vars ISR_CACHE_BUST=...` creates a new revision but uses the same container image as the prior revision. So if the prior image was built from stale source (per the background-then-edit pattern above), the env-var bust gives you a fresh empty cache but the same broken code. To serve new code you need a fresh `--source .` build. (Sprint 32)
- **Payment integrations need a real-device smoke test before ship**: the support-card "DuitNow QR" was broken from Sprint 4.1 until Sprint 32 (~2 months) because nobody scanned it with an actual banking app. The implementation was encoding plain text; a real DuitNow QR is an EMVCo TLV payload with a PayNet-issued merchant ID + CRC16 checksum. Any UI claiming "scan to pay" must pass at least one real bank app's scan check before merging. Automated unit tests can't catch this — only a real scan does. (Sprint 32)
- **Never diagnose Next.js page render via curl alone**: Next.js `redirect()` from a Server Component returns an HTML *shell* page (~500 chars, navigation only) that triggers client-side navigation, NOT an HTTP 30x. curl sees `200 OK` + small body and a careless reader concludes "render failed". Use Playwright (executes JS, follows the redirect) or curl the *canonical* URL directly. If you must inspect via curl, `curl -L` doesn't help (the shell isn't a 30x) — look for the shell pattern (no `<h1>`, no real content) instead of relying on body size. (Sprint 31)
- **Before any production rollback, write the falsifiable hypothesis**: "if I roll back to revision X, the page renders >2000 chars; if it still renders <500, the bug is downstream of code." If the first rollback falsifies the hypothesis, STOP and re-diagnose — don't keep rolling back hoping for a different result. Sprint 17's lesson ("retrospectives that claim work is done aren't proof") has a sibling: "rollbacks that don't fix the symptom mean the diagnosis was wrong, not that you need to roll back further." (Sprint 31)

- Hoisting constants out of loop bodies (regex patterns, sets) is easy to miss during initial development — run a simplification pass after each feature sprint (Sprint 0.3)
- Signal handlers that resolve settings on every fire (e.g. `_get_tracked_models()`) can be silently expensive — cache at module level when the setting doesn't change at runtime (Sprint 0.3)
- f-string logging (`logger.warning(f"...")`) defeats lazy evaluation — always use `%s` style for log messages (Sprint 0.3)
- `%-d` strftime flag is Linux-only — use `f"{d.day}"` for cross-platform no-leading-zero day formatting (Sprint 0.4)
- `google.generativeai` is deprecated — use `google.genai` with client pattern (`genai.Client(api_key=...)`) instead of global `genai.configure()` (Sprint 0.4)
- Django ChoiceField with `required=False` saves empty string `''`, not None — for IntegerField model fields use `TypedChoiceField(coerce=int, empty_value=None)` (Sprint 0.5)
- When redirecting after a model update, double-check the target URL's kwargs match the right model — `pk=instance.fk_id` redirects to the FK's PK, not the instance's PK (Sprint 0.5)
- Setting env vars from bash→PowerShell is unreliable (`$env:VAR` escaping fails). For management commands needing env vars, write a temp Python script that reads `.env` and calls `django.setup()` + `call_command()` — delete after use (Sprint 0.6)
- Database passwords with `?`, `+`, `@`, `#` break `DATABASE_URL` URI parsing — always URL-encode special characters (`%3F`, `%2B`, `%40`, `%23`) in the password portion (Sprint 0.6)
- Test CSV encoding must match what the import command reads — `utf-8-sig` BOM corrupts the first column header when read with `cp1252`. Always use the same encoding in test fixtures as in production code (Sprint 1.1)
- Django ORM `.annotate()` drops the model's Meta.ordering — always chain `.order_by()` after annotate to avoid DRF's UnorderedObjectListWarning (Sprint 1.2)
- Jest `setupFiles` runs BEFORE test framework globals (`expect`, `test`) are available — import `@testing-library/jest-dom` directly in each test file, not via setupFiles (Sprint 1.3)
- Next.js tsconfig uses `jsx: "preserve"` (framework handles transform), but ts-jest needs `jsx: "react-jsx"` — pass inline tsconfig object to ts-jest instead of referencing tsconfig.json (Sprint 1.3)
- Jest config options: `testPathPattern` is CLI-only, use `testMatch` in config; `setupFilesAfterSetup` doesn't exist, use `setupFiles` (Sprint 1.3)
- Django UUIDField `.get(token=value)` raises `django.core.exceptions.ValidationError` (not `ValueError`) when value is not a valid UUID string — always catch `ValidationError` alongside `DoesNotExist` and `ValueError` (Sprint 1.6)
- When Django `post_save` signals create automatic AuditLog entries for tracked models, explicit AuditLog entries in views create duplicates — tests that filter by action/target may get the wrong entry. Filter by a unique field in the `detail` dict (e.g. `changed_fields`) instead of relying on `.first()` (Sprint 1.7)
- Literal Unicode characters (em dash, curly quotes) in HTML email templates render as diamonds in some email clients — always use HTML entities (`&mdash;`, `&ldquo;`) in email HTML, and plain ASCII in subject lines (Sprint 1.9)
- Django `startapp` creates `tests.py` which conflicts with a `tests/` package directory — delete `tests.py` before creating the tests package, and clear `__pycache__` if pytest reports import mismatches (Sprint 2.1)
- `auto_now_add` timestamps can be identical within fast test execution — test ordering via `Model._meta.ordering` instead of queryset comparisons (Sprint 2.1)
- Never bake API keys into stored URLs — store the reference/coordinates and construct the URL at serve time. Key rotation breaks every stored URL otherwise (Sprint 1.10)
- Navigation gaps compound silently — always verify that new page types are reachable from the main entry points (map, search, nav) before closing the sprint (Sprint 1.10)
- Never make HTTP/API calls inside `transaction.atomic()` — the DB connection is held open for the entire API call duration. Move external calls outside the transaction block and use a flag to trigger them after commit (Sprint 2.3)
- Status-gated operations (e.g. DRAFT → SENDING) need atomic conditional UPDATE (`Model.objects.filter(pk=x, status=DRAFT).update(status=SENDING)`) to prevent race conditions — a naive `get()` + check + `save()` allows concurrent requests to both pass the gate (Sprint 2.3)
- `<input type="email" required>` blocks form submission for invalid emails at the browser level — tests that mock API errors must use valid-looking emails, otherwise the mock is never called and the test silently passes or times out (Sprint 2.4)
- `lxml.html.clean` was split into a separate package `lxml_html_clean` — libraries depending on lxml (e.g. trafilatura via justext) will fail with ImportError unless you explicitly add `lxml_html_clean` to requirements (Sprint 2.5)
- gcloud CLI hardcodes Python path — if you upgrade Python versions, set CLOUDSDK_PYTHON env var to the new path (Sprint 2.8)
- Django JSONField __contains doesn't work on SQLite — use vendor-specific branching or LIKE fallback for tests (Sprint 2.8)
- Large sprints (14+ tasks) at high context usage (>85%) risk data loss on crash — push commits frequently and use a branch for safety when the plan is big (Sprint 3.3)
- Supabase Supavisor (connection pooler) holds idle sessions to `test_postgres` that prevent Django's test runner from dropping/recreating the test DB — use `--keepdb` flag or manually terminate Supavisor sessions via `pg_terminate_backend` before running tests (Sprint 3.4)
- When a page route uses `[id]` (numeric PK) vs `[code]` (string), always check which identifier the target page expects before constructing links — mixing up `dun_code` and `dun_id` produces valid-looking but broken URLs (Sprint 3.7)
- Supabase transaction pooler (port 6543) can silently drop sequential writes — use direct connection (port 5432) for bulk operations, and add `connection.close()` after each write as defence (Hansard Backfill)
- When an AI returns empty fields, verify it's actually a write failure before assuming DB issues — Gemini legitimately returns empty `mp_name` when the speaker can't be identified from the quote. Check the right field (`ai_summary` not `mp_name`) to determine if analysis was done (Hansard Backfill)
- Government websites may block specific HTTP methods (HEAD) while allowing others (GET) — when a probe fails, try alternative methods before assuming the resource doesn't exist (Hansard Backfill)
- `gcloud run deploy --set-env-vars` replaces ALL env vars, `--update-env-vars` merges — always use `--update-env-vars` or `--env-vars-file` to avoid wiping existing vars (UI Polish)
- Review-gated APIs (e.g. `review_status=APPROVED`) with no approval workflow create invisible data — if no one is reviewing, nothing shows. Either auto-approve or show pending data (UI Polish)
- Gemini 2.5 Flash thinking tokens consume the output token budget — set `thinking_budget` explicitly (1024 for report-length content) or output will be truncated. `thinking_budget=0` produces degenerate output (Sprint 5.5)
- The `smarty` markdown extension converts `'` to `&rsquo;` HTML entities which display as raw text when tags are stripped but entities aren't decoded — avoid `smarty` unless rendering full HTML (Sprint 5.5)
- `gcloud run deploy --source .` can silently drop env vars set via `--update-env-vars` — always verify critical env vars (BREVO_API_KEY, GEMINI_API_KEY) after every redeployment (Email Infra)
- Silent email failures erode trust — if a confirmation email fails to send, the user should see a warning, not a success message. Never ignore the return value of external API calls that affect user expectations (Email Infra)
- When mocking an AI SDK (e.g. `genai`), also mock the env var guard (`os.environ["GEMINI_API_KEY"]`) — services that check for the key before calling the SDK will early-return and the mock is never reached (Email Infra)
- Google OAuth Playground tokens generated without "Use your own OAuth credentials" checked produce `invalid_grant` — always verify the gear icon settings before authorising (Email Infra)
- Fix source data, not downstream workarounds — when MOE data has inconsistent abbreviations (Ldg vs Ladang), a data migration normalising the source is cleaner than expanding alias matching logic (Data Quality)
- Cloud Run job default timeout is 600s — bulk operations (trigram matching over hundreds of records) need explicit `--task-timeout` increase. Check job logs for timeout errors before assuming code bugs (Data Quality)
- Government website search/filter APIs may ignore parameters entirely — parlimen.gov.my carian.html returns the same results regardless of `takwimnum`, and the calendar page ignores `ssid`. When API filtering fails, fall back to brute-force date range probing (Full Rebuild)
- Gemini analysis at scale hits rate limits around 130-140 calls — build in retry logic or expect to rerun. The 0.5s sleep between calls is not sufficient buffer for sustained bursts (Full Rebuild)
- ISR / cache-control claims must be verified via curl on the deployed revision — `export const revalidate = N` is necessary but not sufficient. Any uncached fetch, dynamic API call, or `next-intl/server.getMessages()` reading cookies/headers can silently override it. Sprint 17 + 21 both hit this; second incident → curl-the-header is now part of the close workflow for any caching change (Sprint 21)
- For caching/ISR sprints, inspect the local `next build` route table (`ƒ` Dynamic vs `●` SSG markers) BEFORE the first deploy. Sprint 21 burned 3 web deploys finding what one local build inspection would have shown (Sprint 21)
- GCP log-based metrics with `valueExtractor` are forced to `valueType: DISTRIBUTION` — INT64 + valueExtractor returns INVALID_ARGUMENT. Cloud Monitoring dashboards using GUI-style aggregation + `pickTimeSeriesFilter` fail with "input cannot be a distribution"; use MQL `fetch ... | align delta(1h) | group_by [metric.X], sum(value) | top N` for guaranteed scalar output (Sprint 21)
- For tasks with a "monitor for N days" component, split into engineering + observation tasks at sprint-plan time. Otherwise the task either gets prematurely closed or stays artificially open (Sprint 21)
- Static/rarely-changing data on high-traffic pages must be fetched server-side with ISR, not client-side per visitor — Googlebot alone can generate hundreds of requests/day, and each client-side fetch hits the database. Match ISR revalidation period to actual data change frequency (Sprint 8.3)
- When multiple commands share a pattern (e.g. hero image handling), copy the working implementation exactly — don't improvise a variation. A misleading variable name (`hero_image_url` holding `bytes`) can hide a bug for weeks until the code path is actually exercised (Sprint 8.6)
- Django `select_related()` fetches full rows including large text fields (WKT polygons, JSON blobs) even when serializers discard them — always audit JOINed models for oversized fields and add `.defer()` for fields not used in serializers or `__str__` (Egress Fix)
- Next.js ISR cache is in-memory + filesystem — both destroyed on Cloud Run scale-to-zero. Either set `minScale=1` or add a persistent cache handler (Redis). Without this, every cold start re-fetches all ISR pages from the database (Egress Fix)
- When two record types share the same `audience_filter`/category string, downstream queries that filter only on that string conflate them. Type vs. audience are independent dimensions — add a first-class `kind` field from the start rather than encoding type into the audience filter (News Digest & Urgent Alert Fix)
- LLM prompts that list positive triggers and negative examples in the same prose block allow keyword pattern-matching to overwhelm the veto. Structure urgency-style classifications as sequential gates (Step 1 trigger match; Step 2 actionability check) and follow up with a second-pass verification call — at ~1 urgent event per month, the verification cost is negligible and catches first-pass false positives (News Digest & Urgent Alert Fix)
- Recurring-period calculations (digest coverage, billing periods) should derive the next start from the previous *period end* stored as data, not from the previous record's `created_at`. Using `created_at` introduces an off-by-one overlap on every cycle (News Digest & Urgent Alert Fix)
- The repo-root `.env` is auto-loaded by `manage.py` and can contain a production `DATABASE_URL` (Supabase pooler). Every local `manage.py` command then writes to prod, not sqlite — "local testing" becomes a silent prod mutation. Before running migrations or data commands locally, either `unset DATABASE_URL` / comment it out, or explicitly verify `python manage.py dbshell` shows the expected DB before writing (News Digest & Urgent Alert Fix)
- Any code path that inspects `DATABASE_URL` to distinguish prod from local must also check whether it's running on Cloud Run (`K_SERVICE` env var set) or with production settings (`DJANGO_SETTINGS_MODULE=sjktconnect.settings.production`). Write the environment-detection bypass FIRST, before the guard logic — otherwise the guard will fire during Cloud Run container startup (which runs `manage.py migrate` against the prod DB, correctly) and fail the deploy. See the initial breakage of the prod-DB guard added 2026-04-23 (Audit & Community Auth)
- At session start, re-read any existing proposal or post-mortem in `docs/proposals/` or `docs/retrospective-*.md` for the area you're working on. On this sprint, the Cloudflare proposal from 6 weeks earlier already described the cross-domain cookie bug that we spent ~6 hours re-diagnosing from three different proximate symptoms (OAuth `org_internal`, incognito third-party cookie block, regular-Chrome `SameSite=Lax`). The answer was already written down (Audit & Community Auth)
- `gcloud run deploy` may exit 0 even when the new revision fails health-check and isn't serving traffic. Always verify with `gcloud run revisions list --service=X --limit=2` and check the ACTIVE column. Exit code alone is not sufficient (Audit & Community Auth)
- `FileReader.readAsDataURL(file)` returns a full `data:image/jpeg;base64,...` string, not raw base64. If the backend expects raw base64 for `b64decode()`, strip the prefix via `.split(",")[1]` before sending. If the frontend also needs to render a preview, keep the raw base64 in state and prepend the prefix in the `<img src={...}>` attribute (Audit & Community Auth)
- Storing a relative URL (`/api/v1/...`) as an image `src` in the DB only works when the frontend and API share a domain. For a frontend on `tamilschool.org` calling an API on `*.run.app`, the browser resolves the relative URL against the frontend and 404s. Always store absolute URLs for cross-service references, sourced from a `BACKEND_URL` setting env-injected at deploy (Audit & Community Auth)
- Cross-domain cookies between frontend (`tamilschool.org`) and API (`*.run.app`) require `SESSION_COOKIE_SAMESITE = "None"` + `SESSION_COOKIE_SECURE = True` to be sent on `fetch()`. Default Django is `SameSite=Lax`, which works for page navigation but not for AJAX to a different origin. Long-term fix is a same-origin reverse proxy (Cloudflare); short-term, `SameSite=None` works but expands CSRF surface — pin `REST_FRAMEWORK["DEFAULT_AUTHENTICATION_CLASSES"] = [SessionAuthentication]` explicitly to preserve CSRF enforcement (Audit & Community Auth)
- OAuth consent screen "Internal" blocks all Google accounts outside the organisation's Workspace with `Error 403: org_internal`. For community-facing apps, switch to "External + Production". With only `openid email profile` scopes, this requires no Google verification and the 100-user cap shown in the UI does not apply (it's for sensitive/restricted scopes only). Name the OAuth consent app after the product, not its original Gmail-feedback use case (Audit & Community Auth)
- When designing UX for a feature whose pattern is well-established on the web (sign-in, claim, share, edit, comment, etc.), look up an actual reference (Google Business profile for "claim", Stripe Dashboard for billing, etc.) before coding the first version. "Look at how Google does it" is a five-minute search and prevents the build-then-redesign cycle that cost ~1 hour on Sprint 11a Phase 3 (Sprint 11a)
- When `gcloud domains verify` (or any gcloud command that opens a browser) opens Search Console / Console.cloud.google.com, the URL inherits the browser's default Google account, NOT gcloud's active account. Always check the signed-in identity in the opened tab before completing any verification flow. If wrong, sign out everywhere and back in with the correct account first; otherwise the verification succeeds but Cloud Run won't accept it (the domain is verified for the wrong project owner). Easy 2-min fix afterwards: add the gcloud account as Owner in Search Console > Settings > Users and permissions (Sprint 11a)
- Next.js `NEXT_PUBLIC_*` env vars are baked into the JavaScript bundle at `npm run build` — Cloud Run runtime env var changes have NO effect on them. To change a NEXT_PUBLIC_ value, rebuild the container image (`gcloud run deploy --source .`); patch the Dockerfile if the value is hardcoded as ENV there. Runtime env vars on Cloud Run only affect server-side code (route handlers, middleware, etc.) (Sprint 11a)
- DKIM CNAME records (e.g. `brevo*._domainkey`) MUST be set to "DNS only" (NOT proxied) in Cloudflare. DKIM is a DNS-level mail-auth protocol, not HTTP traffic — proxying breaks the resolution and silently destroys email deliverability. Cloudflare's UI flags this with a generic warning that sounds like best practice but is wrong for DKIM CNAMEs (Sprint 11a)
- Don't tackle TypeScript strict-mode hygiene during a framework upgrade — they're separate concerns. The framework upgrade should change as little as possible to confirm the framework works; type cleanup is its own sprint with its own scope. On Sprint 11a Phase 4, removing `ignoreBuildErrors: true` exposed pre-existing implicit-any issues that were unrelated to the Next 14→16 upgrade and consumed time without value (Sprint 11a)
- For any Cloud Run service mapped to a custom subdomain (`api.tamilschool.org` etc.) where Cloudflare proxies in front, set the CNAME initially as "DNS only" (grey cloud) so Cloud Run's managed SSL provisioning can verify the CNAME. Once cert is issued (`Ready: True` from `gcloud beta run domain-mappings describe`), flip to Proxied (orange cloud). Set Cloudflare SSL/TLS mode to Full (strict) so end-to-end TLS is verified (Sprint 11a)
- Before adding a new named function or endpoint to a file that already has similar functionality, grep the file for the concept (not just the exact name). `grep -E 'mySuggest|MySuggest|/me/suggest|/suggestions/mine'` would have caught the duplicate `fetchMySuggestions` in Sprint 12's lib/api.ts edit. Especially applies to composition sprints where existing surface is being extended (Sprint 12)
- Beta-version dependencies are acceptable but compat-fragile — lockfile drift from a rebuild can surface previously-hidden regressions. For any beta-version lib (e.g. `next-auth@5.0.0-beta.*`), verify the critical flow (sign-in, auth callback, webhook) after every container rebuild, not just after code changes. Sprint 11a Phase 4 sign-in worked; Sprint 12 frontend rebuild (no auth.ts changes) regressed state-cookie handling. Consider pinning lockfile more aggressively for auth-critical deps (Sprint 12)
- For any auth regression on prod, time-box diagnosis to 30 minutes; beyond that, rollback to the last known-good config immediately. Deeper diagnosis belongs in a dedicated debugging sprint with local repro, not ad-hoc on prod deploys. Sprint 12 spent ~1 hour in a sign-in debug loop over 4 frontend redeploys before settling on the `checks: []` rollback that was already known to work (Sprint 12)
- Django settings child modules must MUTATE parent dicts, not REASSIGN them. Writing `STORAGES = { "staticfiles": ... }` in `production.py` silently wiped the carefully built `STORAGES["default"]` from `base.py`. The same gotcha is well-known for `INSTALLED_APPS`, but it generalises to any composite setting: prefer `STORAGES["staticfiles"] = {...}` (Sprint 13)
- Always set `MEDIA_ROOT` explicitly. With it undefined, Django's `default_storage.location` resolves to whatever `BASE_DIR` quirk applies — in our case, `backend/`, so `image_file.save()` in tests created `backend/schools/<moe_code>/<filename>` directories that landed in `git status`. Set `MEDIA_ROOT = BASE_DIR / "media"` and gitignore that directory (Sprint 13)
- Nullable Django FileField empty rows can be either NULL or empty string `""` depending on how the row was written. `filter(image_file="")` matched 0 of 1500 unmigrated rows; the correct filter is `Q(image_file__isnull=True) | Q(image_file="")` (Sprint 13)
- Multi-API Google calls (Static Maps + Places + Maps JS) must verify EVERY API is enabled on the key's restriction list, not just the obvious one. Re-harvest failed all SATELLITE rows because the backend key was Places-only; Static Maps required a separately-enabled key (Sprint 13)
- When an API endpoint is deliberately trimmed for egress (e.g. `/schools/map/` returns 10 fields not 30), downstream UIs that need the full payload must lazy-fetch on interaction (e.g. on InfoWindow open) rather than expect the trimmed payload to satisfy them. The trim is the correct optimisation; the UI is the place that has to adapt (Sprint 13)
- Perceptual hashing (`imagehash.phash`) is by design colour-tolerant and resize-tolerant — different colours of the same shape, or the same image at different sizes, hash identically. Test fixtures that need distinct pHashes must use actual visual variation (e.g. seeded random pixel grids), not solid colours or differing dimensions (Sprint 14)
- DRF's `SimpleRateThrottle` reads its rate via `THROTTLE_RATES = api_settings.DEFAULT_THROTTLE_RATES` at class-definition time, so `@override_settings(REST_FRAMEWORK={...})` does NOT reach already-imported throttle classes. To temporarily lower the rate in a test, mutate the throttle class's `rate` attribute directly in setUp/tearDown (Sprint 14)
- TypeScript classes that `extends Error` lose their prototype chain when transpiled to ES5, breaking `instanceof CustomError` checks in browsers AND tests. Always add `Object.setPrototypeOf(this, ClassName.prototype)` in the constructor of any custom Error subclass — production fix, not a test workaround (Sprint 14)
- jsdom enforces native HTML form validity (e.g. `<input type="file" required>`) on `fireEvent.click(submitButton)` even when React state already holds the file — because setting `input.files` via Object.defineProperty doesn't propagate to the validity API. Use `fireEvent.submit(form)` directly to bypass the native check (Sprint 14)
- Before any migration that drops a column carrying live user data (`Suggestion.image` BinaryField in Sprint 14), grep for every downstream pointer that was set by the old flow — not just the obvious migration-source field. The Sprint 14 drop missed `SchoolImage.image_url` rows that pointed at `/api/v1/suggestions/<pk>/image/` (an endpoint the same sprint deleted). Generalise: for any "drop X" migration, also search for every URL/path/reference that the old code constructed pointing into X (Sprint 14 hotfix)
- When wiring a new admin UI feature, pre-mortem the role matrix BEFORE shipping: does this work for SUPERADMIN, MODERATOR, school admin, and regular user? Sprint 14's "Make hero" button shipped working only for bound school admins because /dashboard/images was gated on `profile.admin_school` — SUPERADMIN has none. Add a "test with each role" line to local smoke-test plans for any admin-UI sprint (Sprint 14 hotfix)
- New admin entry points should live where related admin actions already live, not in the public-facing chrome. The first hotfix iteration put "Manage images" on the public school header; the second moved it to the edit page where it belongs alongside other admin controls. Default for any admin-only action: nest under an existing admin route, not the public surface (Sprint 14 hotfix)
- When introducing a "consumers should use this instead" property on a model (e.g. `display_url` on top of `image_url`), grep ALL serializers for the underlying field name and migrate every call site in the same PR. Sprint 13 added `display_url` to fix Sprint 13's harvested rows but only updated 2 of 3 serializers; `SchoolListSerializer` was missed and broke the map InfoWindow for migrated rows. The bug surfaced two sprints later (Sprint 15) and required a hotfix deploy. Half-applied fallback patterns are worse than no fallback because they silently shift which call sites are broken (Sprint 15)
- Dual session systems (NextAuth JWT cookie + Django session cookie, etc.) require sign-out to invalidate BOTH sides. NextAuth's `signOut()` only clears the JWT; if the Django session was set during sign-in (e.g. the `GoogleAuthView` handshake writes `request.session["user_profile_id"]`), it outlives the sign-out and any `fetchMe()` after sign-out keeps returning the user. Add a backend logout endpoint that flushes the server session, mark it `AllowAny` + idempotent (so a stale session can self-clear), and `await` it from the frontend BEFORE the client-side `signOut()` call (Sprint 15)
- Components on the same surface that gate visibility on auth state must use a consistent reactivity pattern. Mixing `useSession()` (reactive) with one-shot `useEffect(() => fetchMe().then(...))` (mount-only) leads to bugs where one CTA reacts to sign-in/out and the other doesn't. When adding a new CTA next to an existing one, audit BOTH for (a) reactivity to auth state changes, (b) role-based visibility — they should subscribe to the same source of truth and re-fetch on the same triggers (Sprint 15)
- Before adding any frontend lib that will be imported in test code, check its `package.json` `"type"` field and shipped formats. ESM-only libs (`yet-another-react-lightbox`, etc.) trip Jest's default `transformIgnorePatterns` (which excludes `node_modules`) and either need a `transformIgnorePatterns` exception or integration-only test coverage. Decide upfront which path you'll take so you don't waste 15 min in a Jest config rabbit hole (Sprint 15)
- Visual UI elements that overlap with sibling content (overlays, badges, ribbons, captions on hero images) must be tested against worst-case content quantities, not the median. A caption overlay that looked fine on a school with 3 photos collided with the thumbnail strip on a school with 6 photos. Generalise: any UI dependent on N must test N=1, N=median, N=cap (Sprint 15)
- Public-facing visual polish must be prototyped (Stitch screen) and locally smoke-tested against prod-shape data BEFORE the first deploy. The "max 2 deploys per feature" rule assumes the change was prototyped; iterating polish on prod blows the budget every time. Sprint 15 ate 8 web deploys (94 → 102) chasing visual regressions that would have been visible on `npm run build && npm start` against prod read-only API (Sprint 15)
- Sprint-close test counts must come from actual test-runner output, not from intent. Sprint 15 close recorded "285 frontend tests passing" in CHANGELOG, MEMORY.md, the retrospective, and projects.json — the actual count was 282 pass + 3 fail. The Sprint 15 hotfix added a useSession dep to two components without updating their tests; nobody ran the suite at close. Sprint 16's first task became "fix tests that should have been caught at Sprint 15 close." Future sprint-close commits must include the literal test-runner output (e.g. `Tests: 289 passed, 289 total`) — copying that line is the proof you actually ran the suite (Sprint 16)
